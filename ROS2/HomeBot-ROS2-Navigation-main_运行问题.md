# Q1 Gazebo显示了机器人my_robot的model标签但是无法显示机器人，Rviz2中可以正常显示机器人
## A1：机器人的描述文件中缺乏或者对机器人的物理属性描述有问题
1. *Gazebo 是物理引擎，它不仅看视觉模型（Visual），更看重物理属性*。RViz 只需要 <visual> 标签就能画出机器人，所以 RViz 显示正常。但 Gazebo 如果发现一个 Link 没有 <inertial> 标签，或者质量（mass）被设为 0，它会因为无法计算物理模拟而直接忽略渲染该物体，即Gazebo中需要指明并计算好物体的物理属性（惯性，质量，碰撞参数<collision>等）才能显示物体
2. *物理引擎计算瞬间出错（通常是因为碰撞参数设置不当）*。机器人会在生成的一瞬间获得一个巨大的初速度，直接飞出视野。可以尝试点击Gazebo左侧的model中的my_robot，右键选择move to视野会被强制拉到my_robot处，如果视角切换到了地平面以下或者一片漆黑，说明模型位置漂移了。那么就是可能是我刚刚提到的于坐标设置错误（比如 z 轴坐标太小），它可能掉到了地板（Ground Plane）下面，或者因为受力不均直接被弹到了几公里外。
3. *精细网格文件导致Gazebo计算压力过大*。在 collision 里也使用了精细的网格文件（.stl），Gazebo 的计算压力会很大，有时会出错。 建议： 在 visual 里用复杂的 .stl，但在 collision 里使用简单的 box（盒子）或 cylinder（圆柱）进行简化。如果使用mesh的.dae文件描述机器人的物理特性。Gazebo 计算两个“方块”之间的碰撞只需要几行代码，但计算两个包含几千个三角面片的 .dae 文件的碰撞会导致系统卡顿。数值稳定性：复杂的网格文件容易出现“尖角”，当尖角卡入地面时，物理引擎会产生巨大的排斥力，这就是你之前机器人飞走或掉入地平线下的主要原因之一。
视觉与物理分离：在 ROS 中，最佳实践是：
<visual>：使用复杂的 .dae 或 .stl（为了好看）。
<collision>：使用简单的 box, cylinder, sphere（为了稳定）
4. *可能是shadow的问题导致机器人灰色与环境融为一体了*。
